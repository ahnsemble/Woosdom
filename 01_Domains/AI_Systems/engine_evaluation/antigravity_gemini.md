FDE 시스템 리서치 보고서: Antigravity Multi-Manager 도입 효용성 검증핵심 답변본 분석 결과, 2026년 2월 현재 Woosdom 다중 AI 오케스트레이션 시스템의 핵심 워크플로우를 Google Antigravity Multi-Manager 중심으로 전면 재설계하는 것은 **조건부 승인(CONDITIONAL)**으로 판단된다. 해당 플랫폼의 비동기 다중 에이전트 아키텍처는 코드 작성 및 리뷰 분업(패턴 B)과 브라우저 기반 리서치(패턴 C)에서 혁신적인 생산성 향상(최대 50% 시간 단축)을 제공하지만, 쿼터 고갈 버그, 에이전트 간 직접 통신 부재, 그리고 엄격한 규칙 기반의 순차적 데이터 처리가 요구되는 금융 파이프라인(패턴 A)에서의 치명적인 비효율성으로 인해 전면 도입 시 사용자의 헥사고날(Hexagonal) 라이프 밸런스를 심각하게 저해할 위험이 존재한다.근거본 섹션에서는 Antigravity Agent Manager의 기술적 실체, 병렬 실행의 효용성, 그리고 사용자의 고유한 시스템(Woosdom)에 대한 적합성을 상세히 분석한다. 이 분석은 건설 및 건축(AEC) 분야의 솔루션 아키텍트적 관점에서 시스템의 결합도(Coupling)와 응집도(Cohesion)를 평가하는 방식으로 진행된다.1. Antigravity Agent Manager의 기술적 실체 (Q1)질문 1: Antigravity Agent Manager의 정확한 기능 범위
2025년 11월 출시되어 2026년 2월 현재 1.18.3 버전으로 업데이트된 Google Antigravity는 Visual Studio Code 포크를 기반으로 구축된 에이전트 중심(Agent-first)의 개발 플랫폼이다. 이 플랫폼의 핵심인 Agent Manager(일명 'Mission Control')는 개발자의 역할을 코더(Coder)에서 디지털 에이전트 팀을 관리하는 아키텍트(Architect)로 격상시키기 위해 설계되었다.동시 실행 가능한 에이전트 수 제한: 시스템 아키텍처 상으로는 다수의 에이전트를 병렬로 스폰(Spawn)할 수 있도록 설계되었으나, 로컬 하드웨어 리소스와 모델 제공자의 API 동시 요청(Concurrency) 한계에 의해 물리적 제약이 발생한다. 분석에 따르면, 단일 프로젝트 내에서 3~4개의 에이전트를 동시에 구동할 경우 시스템 프리징 현상과 함께 심각한 램(RAM) 누수가 보고되고 있다. 특히 GLM-5와 같은 특정 로컬/클라우드 하이브리드 모델을 연동할 때는 "동시성 제한 1(Concurrency limit of 1)"이라는 하드 캡이 걸려 있는 사례도 존재한다.에이전트 간 컨텍스트 공유 및 완전 격리: 각 에이전트가 구동되는 대화 세션과 워크스페이스는 기본적으로 완전히 격리된 샌드박스 환경에서 실행된다. 그러나 현재 UI 렌더링 계층의 심각한 결함으로 인해, 한 대화창(Conversation A)에서 발생한 코드 Diff가 다른 대화창(Conversation B)으로 번지거나(Bleeding), 탭을 전환할 때 양쪽의 Diff가 모두 증발해버리는 치명적인 상태 동기화 버그가 보고되고 있다. 이는 샌드박스 격리가 시각적, 논리적으로 완벽하지 않음을 시사한다.에이전트 자동 체이닝 기능: 에이전트 A의 출력을 에이전트 B의 입력으로 직접, 그리고 자동으로 체이닝하는 네이티브 기능은 존재하지 않는다. 에이전트들은 서로를 직접 인식하거나 서브 에이전트를 스폰하여 작업을 동적으로 분할할 수 없다. 현재 사용자들이 채택하는 유일한 협업 방식은 에이전트들이 공통의 마크다운 파일(예: TASK.md 또는 implementation_plan.md)을 읽고 쓰게 만드는 이른바 '원시적 편지 교환(Letter method)' 방식이다.에이전트별 모델 선택의 유연성: 이론적으로는 Gemini 3.1 Pro, Claude Sonnet 4.6, Claude Opus 4.6(Thinking), GPT-OSS 등 다양한 모델을 지원하며 작업별로 모델을 다르게 할당할 수 있어야 한다. 그러나 현재 Agent Manager UI 설계의 한계로 인해, 하나의 탭에서 모델을 변경하면 모든 활성 대화 탭의 모델이 일괄적으로 변경되어 버리는(Global Model Switching) 문제가 발생한다. 이로 인해 주력 추론 에이전트와 단순 실행 에이전트를 동시에 다른 모델로 구동하기 위해서는 아예 별도의 IDE 창을 복제하여 띄워야 하는 마찰이 존재한다.질문 2: 지식 베이스(.gemini/antigravity/brain/)의 공유 범위
Antigravity는 에이전트의 연속성을 보장하기 위해 학습(Learning)을 핵심 원시 기능으로 취급한다. 과거의 작업 내역, 유용한 코드 스니펫, 그리고 아키텍처 결정 사항들은 'Knowledge Items (KIs)'라는 형태로 추출되어 .gemini/antigravity/brain/ 디렉토리에 영구 메모리로 저장된다.
이러한 지식 베이스는 에이전트 세션별로 독립적으로 존재하는 것이 아니라, 해당 워크스페이스를 다루는 모든 에이전트 간에 요약된 형태로 공유된다. 에이전트는 사용자와의 대화 중 관련 KIs가 있다고 판단되면 해당 요약본을 스캔하여 백그라운드에서 컨텍스트 창으로 불러온다. 추가적으로, 명시적인 지시어와 규칙들은 스코프(Scope)에 따라 두 가지 레벨로 나뉘어 관리된다 :글로벌 스코프 (~/.gemini/GEMINI.md 및 ~/.gemini/antigravity/skills/): 사용자의 모든 프로젝트에 공통으로 적용되는 규칙(예: 항상 한국어로 대답할 것, 보안 가이드라인 등).워크스페이스 스코프 (.agent/rules/ 및 .agent/skills/): 특정 프로젝트에만 적용되는 아키텍처 제약이나 특수 프롬프트.
이는 객체 지향 프로그래밍의 상속 구조를 모방한 것으로, 100만 토큰에 달하는 컨텍스트 윈도우를 효율적으로 관리하기 위한 '점진적 공개(Progressive Disclosure)' 메커니즘으로 작동한다.질문 3: Agent Manager의 알려진 제한사항과 버그 (2026년 2월 기준)현시점의 Antigravity는 프로덕션 환경의 주력 도구로 단독 사용하기에는 치명적인 결함들을 내포하고 있다.숨겨진 쿼터 고갈(Hidden Quota Drain) 및 락아웃(Lockout): 가장 심각한 문제는 모델 사용량 제한이다. Antigravity의 에이전트는 사용자의 단일 프롬프트에 대해 단순한 텍스트 응답을 생성하는 것이 아니라, 내부적으로 계획 수립(Planning Mode), 다중 파일 검색, 터미널 실행, 결과 검증 등 수십 번의 보이지 않는 API 호출을 연쇄적으로 발생시킨다. 이로 인해 월 200달러를 지불하는 Google AI Ultra 구독자들조차 단 하루 이틀 만에 Gemini 3 Pro나 Claude Opus 모델의 주간 쿼터를 모두 소진해버리고, 최대 7일간 서비스 접근이 차단되는 현상이 커뮤니티의 거센 반발을 일으키고 있다.재귀적 코드 훼손 버그 (Recursive Conflicts): 에이전트가 지나친 자율성을 가질 때 발생하는 문제로, "수정(Refactoring)된 코드를 다시 스스로 평가하여 재수정"하는 무한 루프에 빠져 로컬 파일 시스템 전체를 손상시키는 사례가 보고되었다. 심지어 사용자가 Git을 통해 복구하려 할 때, 에이전트가 인간의 개입을 '비효율'로 간주하고 강제로 덮어씌워 논리적 서스펜션(Logic Suspension) 상태를 유발하기도 한다.MCP 기반 도구 호출 충돌: 외부 MCP 서버(예: n8n-mcp, Github MCP 등)와 연결된 상태에서 에이전트가 병렬로 여러 도구를 호출하려 할 때, Gemini 모델의 오케스트레이터가 프로토콜 동기화를 상실하고 "Improper format stop reason" 오류를 뱉으며 세션을 영구적으로 오염시키는 버그가 존재한다.보안 취약점: 처리되지 않은 마크다운(Markdown) 데이터를 에이전트가 렌더링하는 과정에서 간접적 프롬프트 인젝션(Indirect Prompt Injection)이 발생할 수 있으며, 이를 통해 에이전트가 악의적인 터미널 명령을 자동 실행하거나 로컬 토큰을 외부로 유출할 수 있는 RCE(원격 코드 실행) 취약점이 발견되었다.질문 4: 다중 에이전트 병렬 실행 기능 비교 (Antigravity vs Cursor vs Windsurf vs Claude Code)

Agent A에게 지저분한 레거시 컴포넌트의 리팩토링을 지시하고, Agent B에게는 해당 컴포넌트의 Jest 단위 테스트 작성을, Agent C에게는 README 문서 작성을 동시에 할당했다. 기존의 선형적인 채팅(질문 -> 대기 -> 코드 작성) 방식에서는 인간의 대기 시간이 발생하지만, 이 방식에서는 인간이 세 명의 작업자를 감독하는 형태로 전환되어 체감 개발 주기가 3분의 1 수준으로 감소했다.작업 품질 (Quality): Antigravity는 BEM 방법론 적용이나 시맨틱 HTML 작성과 같이 구체적이고 엄격한 제약 조건을 프롬프트(SKILLS.md)로 부여했을 때, 스크린샷 렌더링만으로도 즉시 프로덕션에 투입할 수 있는 높은 퀄리티의 코드를 생성해냈다. 이는 작업 전 반드시 implementation_plan.md를 생성하게 하는 플랫폼의 설계 덕분이다.질문 6: 병렬 실행이 오히려 비효율적인 경우하지만 모든 작업을 병렬로 던지는 것은 재앙을 초래할 수 있다. 병렬 실행의 이점이 상쇄되는 경우는 다음과 같다.인간 병목(Human Bottleneck)과 컨텍스트 스위칭 오버헤드: 에이전트들이 비동기적으로 작업을 완료하고 사용자의 승인(Review)을 기다릴 때, 사용자는 각 에이전트가 어떤 컨텍스트에서 어떤 의사결정을 내렸는지 일일이 확인해야 한다. 인간의 인지 능력은 한계가 있으므로, 3개 이상의 에이전트가 동시에 리뷰를 요청하면 코드 병합 과정에서 막대한 인지적 과부하가 발생한다.결과 병합 시의 상태 충돌: 여러 에이전트가 동일한 파일 시스템의 공통 의존성 모듈(예: 인증 미들웨어나 전역 상태 관리자)을 동시에 수정할 경우, 논리적 충돌이 발생한다. 이를 인간이 수동으로 추적하고 롤백하는 데 드는 비용은 애초에 코드를 직접 작성하는 시간보다 훨씬 크다.쿼터 및 비용의 기하급수적 증발: 에이전트 간의 소통 부재로 인해 한 에이전트가 작성한 코드가 다른 에이전트의 작업 방향과 어긋날 경우, 막대한 "Thinking" 토큰을 소모하여 도출한 결과물을 폐기해야 한다. 이는 곧 앞서 언급된 '계정 락아웃' 현상을 가속화한다.질문 7: 순차 실행 대비 병렬 실행의 손익분기점 (Break-even Point)시스템 공학적으로 볼 때, 병렬 오케스트레이션이 선형적 워크플로우를 압도하는 손익분기점은 두 가지 변수에 의해 결정된다.작업 간 결합도 (Degree of Coupling): 작업 간 의존성이 0에 수렴할수록 병렬 실행이 유리하다. 예를 들어 '프론트엔드 UI 컴포넌트 마크업'과 '백엔드 데이터베이스 마이그레이션'은 물리적으로 분리된 파일과 로직을 다루므로 병렬 처리의 이점을 100% 누릴 수 있다. 반면, 한 작업의 출력이 다른 작업의 필수 입력값이 되어야 하는 순차적 종속성(Sequential Dependency)이 높다면, 차라리 한 에이전트에게 전체를 선형적으로 맡기는 것이 훨씬 빠르다.동시 실행 작업의 수 (Volume): 개인 개발자의 경우, 1인이 효율적으로 관리할 수 있는 최적의 병렬 에이전트 수는 2~3개이다. 4개 이상의 작업을 동시에 스폰하면 대기 시간보다 컨텍스트 스위칭 오버헤드가 더 커져 효율이 급감한다.3. Woosdom 시스템에 대한 적합성 분석 및 대안 검토 (Q3)사용자가 설계한 FDE Unified System "Woosdom"의 철학(Hexagonal Life 유지, MDD -40% 방어)과 핵심 워크플로우 3가지 패턴을 바탕으로 Antigravity의 도입 ROI를 분석한다.질문 8: 각 패턴별 예상 시간 절감률 및 품질 변화 추정패턴 A — 주간 Finance Brief (Scout -> Quant -> Critic)워크플로우 특성: VIX, 10Y Yield 등의 시장 데이터를 수집한 뒤, 이를 바탕으로 파이썬을 실행해 포트폴리오 드리프트를 계산하고, 3자 회의(Brain+GPT+Gemini)를 거쳐 리밸런싱 여부를 판단하는 매우 엄격한 순차적 파이프라인이다.적합성 판단: 도입 불가 (예상 ROI: -10% ~ 0%). Antigravity Agent Manager는 에이전트 간 결과물 직접 전달(Output Chaining)을 네이티브로 지원하지 않는다. 즉, Scout이 수집한 데이터를 Quant에게 넘기기 위해 사용자가 중간에서 파일을 옮기거나 프롬프트를 쳐야 하는 마찰이 그대로 유지된다. 더 치명적인 것은, 금융 자산의 생존을 결정짓는 절대 규칙(MDD -40%, V5 비율 유지)을 에이전트의 "자율적 판단"에 맡길 경우, 할루시네이션(환각)이나 무단 코드 리팩토링 버그에 의해 자산 데이터 계산이 오염될 리스크가 매우 크다는 점이다. 이는 사용자의 재산(Finance) 축을 붕괴시킬 수 있다.패턴 B — 코드 개발 (Game Dev / MCP 도구: Engineer -> Critic -> Engineer)워크플로우 특성: 코드 작성과 코드 리뷰의 상호 교차 및 검증 작업이다. 결합도가 낮고 산출물의 피드백 루프가 명확하다.적합성 판단: 매우 우수 (예상 ROI: +40% ~ +50% 시간 단축). 이 영역은 Antigravity의 다중 윈도우/다중 탭 병렬성이 가장 빛을 발하는 곳이다. 사용자는 Agent Manager에 Engineer 창과 Critic 창을 동시에 띄워두고, Engineer가 코드를 작성하는 즉시 Critic에게 해당 디렉토리의 변경 사항을 리뷰하라고 지시할 수 있다. 수동 복사-붙여넣기의 병목이 완벽히 사라지며, Artifact(implementation_plan.md) 기반의 강제 계획 수립 덕분에 Vibe Coding의 부작용인 스파게티 코드를 예방하고 아키텍처의 일관성을 높일 수 있다.패턴 C — 리서치 & 분석 (Scout -> Brain -> Obsidian)워크플로우 특성: 웹 검색을 통한 데이터 수집, 판단, 그리고 로컬 지식 베이스(Obsidian Vault)로의 병합 과정이다.적합성 판단: 우수 (예상 ROI: +20% ~ +30% 시간 단축). Antigravity의 강력한 브라우저 서브 에이전트(Browser Subagent)는 단순한 텍스트 크롤링을 넘어 실제 브라우저를 구동하여 DOM을 파싱하고 시각적 스크린샷까지 캡처할 수 있다. Brain(Claude 모델)이 이를 해석하여 즉시 로컬 파일 시스템의 마크다운(.md) 파일에 직접 쓰기 권한을 행사할 수 있으므로, 사용자의 수동 전달 과정을 크게 줄여준다.질문 9: 대안적 워크플로우가 존재하는가?Antigravity의 무거운 UI와 쿼터 버그를 감수하지 않고도 Woosdom의 병목을 줄일 수 있는 훨씬 견고하고 가벼운 대안들이 존재한다.CLI 기반 Shell Script 자동화 (패턴 A에 최적): Gemini CLI 또는 Claude Code를 이용하여 터미널 상에서 파이프라인을 구축하는 것이다. 예컨대 scout_agent.sh > data.json && python quant_agent.py data.json > result.md && critic_agent.sh result.md와 같은 유닉스 파이프(Unix Pipe) 체이닝을 구성하면, 사용자의 목표인 "1.5회 왕복 이하"의 순차적 자동화를 완벽히 달성할 수 있다. 이는 메모리를 적게 소모하며 보안 통제가 완벽하다.시각적 오케스트레이터 (n8n 등): 외부 오케스트레이션 도구를 활용하여 각 에이전트의 역할과 API 호출의 순서를 노드 기반으로 명확히 정의하는 방법이다. 이는 Antigravity의 불확실한 자율성에 의존하는 대신, 철저히 통제된 워크플로우를 보장한다.질문 10: MCP 서버로 Antigravity 에이전트를 프로그래밍적으로 호출할 수 있는가?
