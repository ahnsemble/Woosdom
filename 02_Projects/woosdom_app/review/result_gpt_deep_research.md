본 리뷰는 첨부된 설계서 **ARCHITECTURE.md v1.1**의 의도(“AI 코딩 에이전트가 정밀 편집 가능한 모듈형 구조”)가 실제 운영·확장·마이그레이션 상황에서 **깨지지 않는가**를 7개 관점에서 비판적으로 검증한 결과입니다. fileciteturn0file0

## 구조적 완결성

### 강점
- **관심사 분리의 방향이 정확합니다.** “데이터 파이프라인(parser.py) → 빌드(build_dashboard.py) → 런타임(Python 데스크톱/브라우저)”로 단계를 분리하고, 산출물(`dashboard_data.json`, `index.html`)을 명시한 점은 레거시(1,600줄 단일 파일, f-string 내부 JS/CSS)에 비해 디버깅·회귀관리·권한분리(누가 어느 파일을 수정하는가)에 유리합니다. 특히 PyWebView/브라우저 병행을 “dev/prod 모드”로 설계한 것은 웹뷰 앱에서 흔한 “브라우저 OK, 앱 Fail” 리스크를 정면으로 다루려는 의도가 명확합니다. citeturn1search8turn2search1  
- **패널 단위 고립(Fault Isolation)을 아키텍처 규칙으로 선언한 점**(IIFE, CSS 네임스페이스, 데이터 미존재 시 무해 렌더링)은, 대시보드처럼 “부분 실패 허용”이 중요한 UI에 잘 맞습니다. 다만 “선언”만으로는 충분하지 않으므로(아래 약점 참고) 이를 강제하는 장치가 함께 있으면 더 강해집니다.
- **“단일 진실 소스(SSOT)”를 데이터로 잡은 점** 자체는 좋은 선택입니다. UI 데이터가 어디서 오는지(Obsidian/기타 파서 → JSON) 단일화하면, UI 템플릿/JS가 더 예측 가능해집니다.

### 약점 / 빠진 부분
- **구조적 SPOF(단일 장애점)가 ‘parser.py → dashboard_data.json’에 집중됩니다.** 설계서가 “dashboard_data.json = 단일 진실 소스”를 강조하지만, 그만큼 파서 실패/스키마 드리프트가 전체 화면을 깨는 확률도 커집니다. 특히 “패널은 데이터가 없으면 조용히 빈 상태로 렌더링” 규칙이 “파서가 조용히 실패했는데도 눈치 못 채는” 문제로 바뀔 수 있습니다(침묵 실패). 이런 경우 운영상 디버깅 시간이 늘어납니다.  
- **‘빌드 타임 렌더링’과 ‘런타임 렌더링(Realtime Layer)’이 방향성 충돌 가능성이 있습니다.** v1.1에서 런타임 `fetch('dashboard_data.json')` 후 `Dashboard.render()`로 패널 모듈의 `render(data)`를 호출하는 구조를 제시했는데, 이는 “Jinja2가 데이터로 완성 HTML을 생성”하는 현재 빌드 파이프라인과 책임이 겹칩니다. 브라우저/웹뷰에서 런타임 `fetch`는 특히 `file://`에서 제한이 자주 발생하며, CORS 오류의 대표 케이스가 “로컬 파일 스킴(file://) 요청”입니다. citeturn0search39turn1search8turn2search1  
- **패널 레지스트리 스키마가 아직 ‘한 가지로 굳지 않았습니다’.** 문서 내 예시가 (a) `id/template/enabled/order` 중심 예시와 (b) `layout/size/collapsible/...`까지 확장된 예시가 혼재합니다. 또한 base.html 예시는 `panel.css`, `panel.js`를 참조하지만 레지스트리 예시에는 그 필드가 없습니다. 이 불일치는 AI 에이전트가 “어느 필드가 정답인지” 혼동해 실패하는 트리거가 됩니다.  
- **품질 게이트(테스트/검증/린트/스키마 검증) 레이어가 ‘훅’만 있고 실행 전략이 비어 있습니다.** JSON Schema를 “추후 검증 가능”으로 적어두었지만, 실제로 언제/어떻게 실패를 감지하고(빌드 실패? 경고?), 어떤 리포트를 남길지(콘솔? 로그 파일?)가 없으면 설계 효용이 크게 줄어듭니다.  

### 제안
- **“렌더링의 단일 책임”을 먼저 고정하세요.** 두 가지 중 하나로 명료화하는 것을 권합니다.  
  - (권장 A) **빌드 타임 렌더링이 기본(현재 설계의 중심)**: `index.html`은 Jinja2로 데이터까지 포함해 완성하고, Realtime은 “부분 갱신/하이드레이션(hydration)”만 담당(예: 특정 DOM만 업데이트).  
  - (옵션 B) 런타임 렌더링이 기본: Jinja2는 컨테이너/레이아웃만 만들고, 패널은 JS가 DOM을 생성(이 경우 템플릿 partial의 성격이 바뀌므로 문서도 크게 바뀌어야 함).  
  런타임 `fetch`는 `file://`에서 문제가 되기 쉬우므로, PyWebView에서는 **내장 HTTP 서버**를 쓰거나(상대경로 로드 시 자동/옵션) citeturn2search1turn1search0turn1search8, JS API 브릿지로 Python이 JSON을 전달하는 모델(예: `window.pywebview.api.get_data()`)도 함께 고려하는 게 안전합니다. citeturn1search12turn2search1  
- **레지스트리 JSON에 대한 ‘정식 스키마’를 문서에 박아 넣으세요.** 필드(필수/선택), 기본값, 정렬 규칙(order 충돌), css/js 매핑 규칙(명시 필드 vs 규칙 기반 자동 탐지)을 단일 버전으로 확정하면 AI 에이전트 작업 성공률이 올라갑니다(“모호함 제거”는 에이전트 운영에서 가장 큰 레버입니다).  
- **침묵 실패 방지를 위한 최소 관측성(Observability) 추가**:  
  - parser 단계: “파서별 성공/실패/생성 키 목록”을 stdout에 표준 포맷으로 출력(또는 로그 파일).  
  - build 단계: “패널별 포함 여부 + 링크/인라인된 리소스 목록”을 출력.  
  - 런타임: 패널 초기화 실패를 잡아 “해당 패널만 degraded 표시(작은 배지/에러 영역)”하도록 최소 try/catch 및 에러 버스(Event) 제공.

## 확장성 검증

### 강점
- **확장성 체크리스트를 ‘파일을 건드리지 말아야 한다’까지 명문화한 점**은 매우 강합니다. 이런 형태는 실제로 설계 품질을 높입니다(“무엇을 바꾸지 않을 것인가”가 명시되면, 구조가 그 요구를 견디도록 설계가 정교해집니다).  
- **‘패널 추가’의 반복 동작을 레지스트리 기반으로 표준화**한 것은, 대시보드류(홈랩 대시보드 포함)에서 검증된 패턴과 유사합니다. 예를 들어 entity["organization","Homer","static dashboard project"] 는 단일 YAML 설정 파일 기반으로 동작하는 정적 대시보드 스타일을 채택하고 있습니다. citeturn7search5turn7search1  

### 약점 / 빠진 부분
- **체크리스트 10개 시나리오 중 “현실적으로 깨질 가능성이 높은 항목”이 있습니다.**
  - **(1) 새 패널 추가**: “build_dashboard.py / core.js / base.html은 절대 건드리지 않는다”는 목표는 좋지만, 레지스트리가 `css/js`를 어떻게 연결하는지(필드? 규칙? 파일 존재 자동 탐지?)가 확정되지 않으면, 새 패널이 스타일/스크립트를 로드하기 위해 결국 base.html 또는 빌드 로직 수정이 필요해집니다(현재 예시 간 불일치가 그 신호입니다).  
  - **(6) 새 데이터 소스 = parsers/ 파일 1개 + registry 1줄**: 파서 플러그인 구조는 강력하지만, 실제로는 “공통 유틸/인증/레이트리밋/캐시/오류 처리/스키마 추가”가 필요해져 registry 1줄로 끝나지 않는 경우가 잦습니다. 특히 외부 API(GitHub 등)를 붙이면 장애·지연·키 관리가 파서 레이어에 유입됩니다.  
  - **(10) 실시간 업데이트 전환**: `setInterval` 한 줄로 끝내려면, 런타임 데이터 로딩이 가능한 실행 환경이 전제입니다. 브라우저에서 `index.html`을 파일로 직접 여는 경우 `fetch`가 `file://` 때문에 막히는 사례가 널리 보고되어 있으며, CORS는 HTTP/HTTPS 스킴이 아닌 요청을 차단하는 오류 유형이 존재합니다. citeturn0search39turn1search8turn1search0  
- **빠진 확장 시나리오(중요)**  
  - **크로스-패널 상호작용**: 예) “헤더 필터/검색이 여러 패널을 동시에 필터링”, “한 패널 클릭이 다른 패널 하이라이트/드릴다운”, “Quick Launcher가 패널 상태를 바꿈”. 이런 요구가 생기면, “패널은 완전 독립” 규칙이 오히려 개발을 어렵게 만들 수 있습니다. 최소한의 이벤트 버스/공유 상태 계약이 필요합니다.  
  - **리소스/성능 예산이 있는 패널**: Canvas(픽셀 에이전트)처럼 CPU/GPU를 쓰는 패널은 “보이지 않을 때 중지”, “프레임레이트 제한”, “모바일/저사양에서 비활성화” 같은 런타임 정책이 필요해집니다(레지스트리에 enabled만으로는 부족).  
  - **버전 호환(데이터 스키마 버전업)**: `dashboard_data.json`이 진실 소스라면, 스키마 버전(예: `schema_version`)과 마이그레이션/폴백 전략이 확장성의 핵심입니다.

### 제안
- **“패널 1개 추가 = 파일 3개 + registry 1줄” 공식을 유지하려면, ‘규칙 기반 자동 탐지’를 강제하는 편이 더 낫습니다.**  
  예: `id = foo`면 빌드가 자동으로  
  - template: `partials/foo.html`  
  - css: `css/panels/foo.css`(존재하면 포함)  
  - js: `js/panels/foo.js`(존재하면 포함)  
  이렇게 “존재 여부 기반”으로 처리하면 레지스트리의 스키마가 간결해지고, AI 에이전트가 필드 실수로 깨뜨릴 확률이 줄어듭니다. (반대로 레지스트리에 css/js 경로를 명시하는 모델은, 경로 오타가 실패의 주요 원인이 됩니다.)  
- **확장성 체크리스트에 ‘교차-패널 이벤트’와 ‘성능 예산’을 추가**하세요. 대시보드/위젯 시스템에서는 장기적으로 이 둘이 유지보수 비용의 대부분을 차지하는 경향이 있습니다(특히 Canvas/차트/네트워크 호출이 들어가면).  
- **Realtime은 PyWebView 런타임 제약(HTTP server / file://)을 설계에 포함**해야 합니다. PyWebView는 `file://` 사용을 “제약/한계가 많아 권장하지 않는다”고 문서에서 언급하고, 로컬 정적 파일 서비스용 내장 HTTP 서버 사용 방식을 제공합니다. citeturn1search8turn2search1turn1search0 이 전제를 만족시키면 checklist (10)의 실현 가능성이 크게 올라갑니다.

## 기술 선택 타당성

### 강점
- **Jinja2 선택은 “Python으로 빌드(단일 커맨드) + 템플릿 상속/부분 include/매크로” 요구에 정합적입니다.** Jinja는 템플릿 상속·include·매크로·확장(extensions) 같은 기능을 공식 문서에서 핵심 기능으로 제시합니다. citeturn0search0turn0search8turn0search28  
- **Vanilla CSS + CSS Variables는 테마/토큰 중심 설계에 적합합니다.** CSS 커스텀 프로퍼티는 표준 기능으로 자리잡았고, :root에 토큰을 두고 테마별로 오버라이드하는 패턴은 정석적인 접근입니다. citeturn2search2turn2search18turn2search6  
- **Vanilla JS(ES5+) 선택은 PyWebView의 다중 엔진 환경을 고려하면 보수적으로 안전한 편입니다.** PyWebView는 OS/렌더러별 차이를 갖고, 문서/변경로그에서도 호환성 이슈를 지속적으로 다루고 있습니다. citeturn0search6turn0search14turn1search8  

### 약점 / 빠진 부분
- **Jinja2의 “자동 이스케이프”는 설정에 따라 달라집니다.** Jinja는 기본적으로 autoescape를 명시적으로 구성하는 것을 권장하며, `select_autoescape`를 통해 확장자 기반으로 켤 수 있습니다. citeturn5search1turn5search2turn5search9 설계서에서 이 부분을 “기본 제공”처럼 기대하면(특히 `.html`이 아닌 `.j2` 같은 확장자를 쓰게 되면) 보안/표현 동작이 달라질 수 있습니다.  
- **대안 템플릿 엔진과의 비교에서 핵심 축이 빠져 있습니다.**
  - `string.Template`는 표준 라이브러리이지만, “단순 치환”에 적합한 간소화된 템플릿이며(교육/문서에서도 ‘단순한 문법’ 강조) citeturn5search8turn5search4, 상속/매크로/파트 분해 같은 요구를 충족하기 어렵습니다.  
  - Mako는 Python 코드에 더 가깝게 템플릿을 작성할 수 있어 강력하지만(“embedded Python” 성격), 이로 인해 템플릿 내 로직 과다/보안(이스케이프 정책) 리스크가 커질 수 있습니다. 환경 전역의 자동 이스케이프 메커니즘이 약하다는 보안 도구 문서도 있습니다. citeturn6search10turn6search4  
  - Nunjucks는 Jinja 문법에 영감을 받은 JS 템플릿 엔진으로 상속/블록을 제공하지만, “Python 빌드 스크립트 한 줄”이라는 현재 빌드 철학과는 결이 다릅니다(노드 런타임/패키징 부담). citeturn2search3turn2search19  
- **“No React, No Webpack”의 장기 리스크는 ‘기술’이 아니라 ‘운영 도구 부재’에 있습니다.** 번들링/디펜던시 고정(lock)/모듈 경계 검사/트리쉐이킹/린트 등이 자동으로 붙지 않기 때문에, 시간이 지날수록 “규칙을 인간이 지켜야 하는” 비용이 커질 수 있습니다.

### 제안
- **Jinja2를 유지하되, autoescape/undefined 정책을 ‘설계서에 명시’하세요.** 예:  
  - autoescape: `select_autoescape(['html','htm','xml'])` 사용 citeturn5search1  
  - undefined: 누락 변수는 빌드 실패(StrictUndefined) 또는 경고(패널 단위) 등 정책 결정  
  이 2개가 “AI 에이전트가 작업 후 빌드가 깨지는 패턴”을 크게 줄입니다(조기 실패가 더 싸게 끝납니다).  
- **차트/외부 라이브러리 도입(Chart.js 등)은 ‘오프라인/번들’ 관점에서 문서화가 필요합니다.** Chart.js는 MIT 라이선스임을 문서에서 명시하고 citeturn4search3turn4search24, 설치 경로에 따라 배포물(dist) 취급이 달라질 수 있습니다. citeturn4search17  
  - 개발: CDN 사용(간편)  
  - prod/py2app: **로컬 vendoring**(인터넷 없이도 동작) + 빌드 스크립트에서 인라인/복사  
  같은 운영 정책을 정해두면 “앱에서만 차트가 안 뜬다”류 이슈를 줄일 수 있습니다.
- **유사 프로젝트 성공 사례 관점에서의 결론**: 개인/홈랩 대시보드 영역에서는 “단일 설정 파일(Registry/YAML) + 정적 자산 + 가벼운 프론트”가 흔한 성공 패턴입니다. Homer(정적 + YAML), Heimdall/Organizr(웹앱이지만 ‘대시보드/탭’ 구성 중심), macOS 위젯 엔진인 Übersicht(위젯 폴더에 JS/CSS 기반 위젯 배치)는 “구성-중심, 파일-기반 모듈화”라는 점에서 설계 방향을 지지하는 사례로 볼 수 있습니다. citeturn7search5turn7search1turn7search6turn7search22turn1search14turn1search34  

## AI 에이전트 친화성

### 강점
- **현재 pain point(“긴 단일 파일 + f-string 내부 JS/CSS”)에 대해, 제안 구조는 정면 해소입니다.** AI 코딩 에이전트의 실패 패턴 중 하나는 “컨텍스트가 너무 길어 핵심을 놓치거나, 문자열/이스케이프 경계에서 오작동”하는 것입니다. 긴 컨텍스트에서 중요한 정보가 가운데에 있을 때 성능이 떨어지는 현상(“lost in the middle”)은 학술적으로도 보고되어 있습니다. citeturn3search2turn3search5  
  파일을 분리하고(HTML/JS/CSS를 네이티브 파일로) 경계를 명확히 하는 것은, 에이전트가 수정 범위를 좁히고 실패 비용을 낮추는 데 직접적으로 기여합니다.
- **“절대 건드리면 안 되는 파일”을 명시한 체크리스트는 에이전트 지시문으로 매우 강력합니다.** Claude Code가 프로젝트 규칙/워크플로우 문서화(예: CLAUDE.md)를 강조하는 것도 “명확한 제약이 성능을 만든다”는 맥락과 맞닿아 있습니다. citeturn3search12turn3search37  
- **Codex/Claude와 같은 에이전트형 도구는 ‘프로젝트 구조/관례에 적응’한다고 명시합니다.** 즉, 구조가 곧 성능 변수입니다. citeturn4search1turn4search4turn4search19  

### 약점 / 빠진 부분
- **새로운 실패 모드가 생깁니다: “다중 파일 정합성” 실패.**  
  - 패널 id, CSS 네임스페이스, DOM id, JS 등록 id가 조금씩 어긋나도 빌드는 통과하지만 런타임에서만 깨질 수 있습니다.  
  - 레지스트리(순서/활성화)와 파일 존재(템플릿/자산) 사이 불일치가 자주 발생합니다.  
- **패널 독립성 규칙이 오히려 ‘공유 기능’ 작업을 어렵게 만들어, 에이전트가 core.js에 과도하게 코드를 쌓는 역효과가 날 수 있습니다.** 결과적으로 core.js가 “새로운 1,600줄 단일 파일”이 되는 경로가 열려 있습니다(핵심 위험).  
- **PyWebView 브릿지(JS API) 초기화 타이밍/가용성 같은 런타임 특이점은 에이전트가 자주 밟는 지뢰입니다.** JS API가 언제 주입되는지/어떤 이벤트에서 안전한지 문제가 이슈로 다뤄진 사례가 있습니다. citeturn1search24turn1search12  

### 제안
- **AI 에이전트용 ‘작업 단위’ 자체를 아키텍처에 포함시키세요.** 다음 3가지는 투자 대비 효과가 큽니다.
  - **Panel Scaffold(생성기) 제공**: `python3 build_dashboard.py --new-panel foo`로 (1) partial (2) css (3) js (4) registry 1줄을 자동 생성. 사람이든 에이전트든 “정합성” 실수를 크게 줄입니다.  
  - **Registry/파일 정합성 검증을 빌드에 포함**: 레지스트리의 panel id마다 “템플릿 파일 존재 여부, css/js 존재 여부(규칙 기반이면 자동), 중복 order 탐지”를 수행하고 실패/경고를 분리합니다.  
  - **에이전트 지시문(프로젝트 규칙 파일) 추가**: Claude Code는 프로젝트 규칙 문서가 성능에 중요하다고 안내합니다. citeturn3search12turn3search37 여기에 “한 번에 한 패널만 수정”, “core.js 수정은 승인 필요”, “레지스트리 스키마 준수” 같은 규칙을 넣으면 안정성이 올라갑니다.  
- **에이전트가 ‘길게 보고 고치기’보다 ‘찾고/검증하며 고치기’에 강해지도록**: SWE-agent 같은 연구는 “에이전트-컴퓨터 인터페이스(검색/탐색/편집/실행)” 설계가 성능에 영향을 준다고 보고합니다. citeturn3search4turn3search14 즉, 구조 + 검증 루프(빌드/스냅샷/비주얼 리그레션)를 결합하는 것이 핵심입니다.

## 마이그레이션 리스크

### 강점
- **Phase를 쪼갠 전략 자체는 합리적입니다.** 특히 “Phase 1에서 골격(디렉토리/분리/빌드러너)만 만들고 동일 결과를 검증”은 레거시 리팩터링에서 정석에 가깝습니다.  
- **dev/prod(링크 vs 인라인) 분리도 마이그레이션 리스크를 줄입니다.** 개발 중에는 디버깅을, 앱 번들링에서는 안정 로드를 우선하는 선택이기 때문입니다. citeturn1search8turn2search1  

### 약점 / 빠진 부분

- **Realtime Layer를 Phase 1~2에서 ‘기반만 깔아놓는다’는 것이 오히려 위험할 수 있습니다.** 기반이 곧 아키텍처 방향을 강제하기 때문에, 앞서 언급한 “빌드 vs 런타임 렌더링 충돌”을 먼저 결정하지 않으면 중간에 재작업이 커질 수 있습니다.

  실무적으로는:
  - 동일 입력(동일 `dashboard_data.json`)에 대해 “구버전 index.html 출력”과 “신버전 출력”을 저장  
  - (가능하면) 헤드리스 브라우저로 렌더링 스크린샷 비교(비주얼 회귀)

  이런 형태가 Phase 1~2에서 특히 강력합니다.
- **롤백은 “구버전 빌더를 유지한 채 병행 출력”이 가장 안전합니다.** Phase 1~2 동안은 `index_old.html`, `index_new.html`을 동시에 생성해 비교 가능하게 하고, 앱 런타임에서 토글(또는 커맨드라인 플래그)로 선택하게 하면 “되돌릴 수 있는 안정감”이 생깁니다.

### 제안
- **PyWebView 런타임 Best Practice를 설계서에 ‘강제 조건’으로 승격**하세요.

  - 데이터 소스(파일/스크립트/API)

### "이대로 진행해도 되는가?"에 대한 명확한 답변
**조건부로 “진행해도 됩니다.”**  
다만 **Phase 1 착수 전에** (1) 렌더링 책임의 단일화, (2) 레지스트리 스키마 일관화 + 빌드 검증, (3) Golden Master 회귀 테스트를 먼저 확정해야 합니다. 이 3가지를 선행하면, 이후 Phase 1~4는 설계서의 의도대로 “AI 에이전트가 파일 단위로 정확히 수정”하는 구조로 수렴할 가능성이 높습니다. 반대로 이 3가지 없이 바로 들어가면, v1.1의 확장 레이어 때문에 “구조는 나눴는데 전체가 더 헷갈리는” 상태로 빠질 위험이 큽니다(특히 Realtime).
