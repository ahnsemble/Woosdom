# Premise Audit Protocol (전제 감사 프로토콜)
*brain_directive.md에 추가할 섹션*
*작성: 2026-02-15*
*트리거: SCHD 프록시 맹점 — Phase 3~10까지 인덱스 원본 데이터 존재를 탐색하지 못한 구조적 실패*

---

## 목적

Brain은 **한 턴 안에서의 자기 점검(단일 메타인지)**은 가능하지만, **턴을 넘어서는 재귀적 메타인지(Recursive Meta-Cognition)**는 구조적으로 불가능하다. 과거 Phase에서 세운 전제가 컨텍스트에 "결론"으로만 남고, "왜 그 결론에 도달했는지"와 "어떤 대안을 기각했는지"가 소실되기 때문이다.

이 프로토콜은 **Phase 전환 시점에 강제로 전제를 재검토**하여, 같은 맹점을 반복하는 것을 방지한다.

---

## 발동 조건 (Trigger)

아래 상황 중 하나라도 해당하면 Premise Audit을 **필수** 실행한다:

1. **Phase 전환 시점** — 새로운 Phase 작업지시서를 작성하기 직전
2. **3회 이상 동일 전제 재사용** — 같은 가정이 3개 이상의 Phase에서 변경 없이 사용될 때
3. **사용자의 "왜?" 질문** — 사용자가 기존 전제에 대해 의문을 제기할 때
4. **외부 엔진 불일치** — 3자 회의에서 GPT/Gemini가 Brain의 전제 자체를 부정할 때
5. **신뢰도 70% 미만 판정** — 이전 Phase 결론의 신뢰도가 70% 이하일 때

---

## 실행 절차 (5단계)

### Step 1: 전제 추출 (Premise Extraction)
현재 Phase에서 **"당연하다고 전제하고 있는 것들"**을 전부 나열한다.

```
체크리스트:
- [ ] 데이터 전제: 어떤 데이터 소스를 쓰고 있는가? 더 나은 소스가 있는가?
- [ ] 방법론 전제: 왜 이 방법을 쓰는가? 대안은 검토했는가?
- [ ] 도구 전제: 왜 이 도구/엔진인가? 다른 도구가 더 적합한가?
- [ ] 범위 전제: 탐색 공간을 임의로 좁히고 있지 않은가?
- [ ] 시간 전제: "이전에 결정했으니까"로 넘어가는 항목이 있는가?
- [ ] 프록시 전제: 대리 지표를 쓰고 있다면, 원본 데이터가 존재하는가?
```

### Step 2: 기원 추적 (Origin Tracing)
각 전제가 **언제, 왜 도입되었는지** 추적한다.

```
| 전제 | 도입 시점 | 당시 근거 | 이후 재검증 여부 |
|------|----------|----------|:---------------:|
| (예시) SCHD→VIG 프록시 | Phase 3 | "VIG가 가장 유사" | ❌ 미검증 |
```

재검증된 적이 없는 전제 → Step 3으로.

### Step 3: 대안 탐색 (Alternative Search)
각 미검증 전제에 대해 **최소 2개의 대안**을 탐색한다.

```
| 전제 | 대안 1 | 대안 2 | 탐색 방법 |
|------|--------|--------|----------|
| (예시) SCHD→VIG | 인덱스 원본 (^DJUSDIV) | SCHD 출시 후 구간만 사용 | 웹 검색 + yfinance |
```

대안이 현재 전제보다 우월하면 → Step 4로.

### Step 4: 영향도 평가 (Impact Assessment)
전제를 변경했을 때 **과거 결론이 얼마나 흔들리는지** 평가한다.

```
영향도 수준:
- 🟢 낮음: 결론 변경 없음 (수치가 ±5% 이내로 변동)
- 🟡 중간: 순위가 변동하지만 Top 3 내 유지
- 🔴 높음: 최종 선택이 바뀔 수 있음
```

🔴 판정 → 해당 전제를 변경하고 재실행을 권고.

### Step 5: 감사 기록 (Audit Log)
결과를 `premise_audit_log.md`에 기록한다.

```markdown
## Premise Audit — Phase [N] → Phase [N+1]
*실행일: YYYY-MM-DD*

### 발견된 전제 [총 X개]
| # | 전제 | 도입 | 재검증 | 대안 존재 | 영향도 | 조치 |
|---|------|------|:------:|:---------:|:------:|------|
| 1 | ... | Ph.3 | ❌ | ✅ | 🔴 | 변경 |
| 2 | ... | Ph.5 | ✅ | — | — | 유지 |

### 변경 실행
- 전제 #1: [기존] → [변경] — 근거: ...

### 미조치 (의도적 유지)
- 전제 #2: [유지 근거]
```

## Brain에게 주는 지시

> **모든 Phase 전환 시, 새 작업지시서를 작성하기 전에 반드시 이 Premise Audit을 먼저 실행하라.**
> 
> "이전 Phase에서 결정한 거니까"는 전제를 유지하는 근거가 될 수 없다.
> 전제를 유지하려면 **"지금 다시 봐도 이게 최선이다"**라는 능동적 확인이 필요하다.
> 
> Audit 결과 변경 사항이 없더라도, "검토했고 유지한다"는 기록 자체가 가치 있다.
> **기록되지 않은 유지는 관성이고, 기록된 유지는 판단이다.**
