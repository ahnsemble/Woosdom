# Systematic Debugging — 4-Phase Root Cause Protocol

## 핵심 원칙 (superpowers 기반)

1. **추측 금지.** "아마 이거겠지" 하고 코드 수정부터 하는 것은 안티패턴.
2. **증거 먼저.** 에러 메시지, 로그, 재현 스텝을 수집한 뒤에 가설 수립.
3. **한 번에 하나만.** 여러 수정을 동시에 적용하면 원인 특정 불가.
4. **되돌리기 준비.** 수정 전 상태를 git stash 또는 백업으로 보존.

## 4-Phase Protocol

### Phase 1: 증상 수집 (Gather)
```
□ 에러 메시지 전문 복사 (스택 트레이스 포함)
□ 재현 스텝 정리 (1. 이걸 하면 2. 이게 나옴)
□ 언제부터 발생? (마지막으로 정상 작동한 시점)
□ 최근 변경사항 확인 (git diff, git log -5)
□ 환경 정보 (Node/Python 버전, OS, 의존성 버전)
```

### Phase 2: 가설 수립 (Hypothesize)
```
□ 증거 기반으로 가능한 원인 3개 이하 나열
□ 각 가설에 "이거라면 X가 참이어야 한다" 검증 조건 명시
□ 가설 우선순위: 가장 검증 비용 낮은 것부터
```

### Phase 3: 격리 검증 (Isolate & Verify)
```
□ 가설 1개씩 순서대로 검증
□ 검증 방법: console.log / breakpoint / 최소 재현 케이스
□ 검증 결과 기록: 가설 확인됨(✅) / 기각됨(❌)
□ 확인된 가설 → Phase 4로. 전부 기각 → Phase 2 재수행
```

### Phase 4: 수정 & 회귀 확인 (Fix & Regress)
```
□ 수정은 최소 범위로 (관련 없는 리팩토링 금지)
□ 수정 후 원래 재현 스텝으로 동일 테스트
□ 기존 테스트 스위트 전체 실행 (회귀 확인)
□ 수정 내용 + 원인을 커밋 메시지에 기록
```

## Brain → Hands 디버깅 위임 템플릿

Brain이 `to_[engine].md` (엔진별 전용 파일)에 디버깅 태스크 작성 시:

```markdown
### T1: 증상 수집
- **작업:** [에러 메시지/로그/재현 스텝] 수집
- **검증:** `에러 전문 + 재현 스텝 + git log -5 기록`
- **예상:** ~3min

### T2: 가설 수립 + 격리 검증
- **작업:** 가설 최대 3개 → 검증 비용 낮은 순으로 테스트
- **검증:** `from_[engine].md에 가설별 ✅/❌ 기록`
- **예상:** ~5min

### T3: 수정 + 회귀 확인
- **작업:** 확인된 원인에 대해 최소 범위 수정
- **검증:** `[테스트 명령어] — pass + 원래 에러 미발생`
- **예상:** ~3min
```

## 금지 사항

| 절대 금지 | 이유 |
|----------|------|
| Phase 1 건너뛰고 바로 코드 수정 | 잘못된 수정 → 새 버그 유발 |
| 한 번에 여러 파일 동시 수정 | 원인 특정 불가 |
| "works on my machine" 선언 | 환경 차이 미확인 |
| 에러 메시지 읽지 않고 추측 | 90% 시간 낭비의 원인 |
| git stash/backup 없이 대규모 수정 | 롤백 불가 |

## Hands가 막혔을 때 (Escalation)

3회 이상 가설 기각 시:
1. 현재까지 수집된 증거 + 시도한 가설 목록을 `from_[engine].md`에 기록
2. Brain에 에스컬레이션 → Brain이 새 가설 방향 제시 또는 다른 엔진 라우팅
3. **절대 같은 접근을 반복하지 않음**
